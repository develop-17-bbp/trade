# ACT GPU-box remote monitoring via Cloudflare Tunnel

The `START_ALL.ps1` ships with a "quick tunnel" (ephemeral `.trycloudflare.com`
URL that rotates on every restart). That's fine for dev but hostile for remote
ops — you can't bookmark it and anyone who spots a URL in a log can hit it.

This directory holds a **named tunnel** replacement:

  * Stable hostnames: `grafana.<your-domain>`, `api.<your-domain>`
  * Persists across restarts (runs as a Windows service)
  * Sits behind **Cloudflare Access** (Zero Trust) — email-OTP / SSO login
    required before any request touches the GPU box
  * Only Grafana and the ACT API server are exposed. Prometheus / Tempo /
    Redis / OTel stay localhost-only (see `infra/docker-compose.yml`).

## One-time setup

From an elevated PowerShell on the GPU box:

```powershell
# 1. Pull latest code
git pull origin main

# 2. Run the setup script — it guides you through login, create, DNS, service install
./infra/cloudflared/setup_tunnel.ps1 -Domain yourdomain.com
```

The script will:

1. `cloudflared tunnel login` — opens a browser, you pick your domain and authorize
2. `cloudflared tunnel create act-gpu` — generates UUID + credentials JSON
3. Renders `config.yml` from `config.yml.template` using the UUID and your domain
4. `cloudflared tunnel route dns` for each hostname (creates CNAME records)
5. `cloudflared service install` — registers it as a Windows service (auto-starts)

## Zero Trust — REQUIRED before sharing the URLs

The tunnel itself has no authentication. The API server uses `DASHBOARD_API_KEY`
and Grafana has its own login, but defense in depth demands a layer in front:

1. https://one.dash.cloudflare.com/ → Access → Applications → Add
2. "Self-hosted" → Application domain: `grafana.<yourdomain>`
3. Policy: `Include → Emails in: your@email` → Allow
4. Repeat for `api.<yourdomain>`

Session duration: 24 hours is reasonable. After that you re-auth with the same
SSO / email OTP. This gives you:

  * Rotatable access (revoke the email policy, done)
  * Audit log of every login (CF dashboard)
  * Brute-force protection (CF rate-limits OTPs)
  * Geo-blocking / device-posture if you want to add those later

## Day-to-day operations

```powershell
# Check tunnel health
cloudflared tunnel info act-gpu

# Restart the service (e.g. after editing config.yml)
Restart-Service cloudflared

# View tunnel logs
Get-EventLog -LogName Application -Source cloudflared -Newest 20

# Poll the process-monitor endpoint
curl.exe -H "x-api-key: $env:DASHBOARD_API_KEY" https://api.yourdomain.com/api/v1/system/processes
```

## What gets exposed

| Hostname | Target | Auth layers |
|---|---|---|
| `grafana.<domain>` | `localhost:3000` | CF Access → Grafana admin login |
| `api.<domain>` | `localhost:11007` | CF Access → `x-api-key` header |

Everything else (Prometheus :9090, Tempo :3200, Redis :6379, OTel :4317/:4318)
stays on `127.0.0.1:*` inside `infra/docker-compose.yml` and is **never**
reachable through the tunnel.

## Rollback

```powershell
# Stop and remove the service
cloudflared service uninstall

# Delete the tunnel (removes CF-side config; leaves your cert.pem)
cloudflared tunnel delete act-gpu
```
